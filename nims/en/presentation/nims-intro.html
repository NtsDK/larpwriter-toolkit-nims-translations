Concept

During making LARP game story master defines character goals, values and motivations. If story master gives instructions like "you are stupid, you hate king, you want to merry princess" then players will ask many questions or just ignore instructions. Thats why it is better when player receive not instructions but stories where his character participated. These stories gives information about pregame events, character behavior, estimations, thoughts etc. Also each story looks different for each story character because no one has full information and has subjective story vision. Writing handouts this way is time-consuming but it gives good result.

To solve this task we created Story Master Toolkit NIMS. In this presentation we will tell you how we describe LARP story making process with NIMS.

Terminology

  • Character – LARP game character.

  • Story – series of connected events which involves some number of characters.

  • Master story – story master version of how story actually happen.

  • Story adaptation – character subjective vision of story events.

  • Handout – player document with character information, story adaptations and some extra information.

Step 1. Master story

First of all you need to invent a story. Thats all. You don't need to divide it in parts, create motivations, add extra details. Just invent story how it happens. This story is a master story.

Example

Ring keeper

Gandalf comes back from a long journey to Frodo and tell him about the One Ring and about black riders. Sam overhear Gandalf's story. Gandalf catches Sam and sends Sam and Frodo to Bree. On the way they meet Merry and Pippin and take them along.

Step 2. Master story processing

Lets begin work with master story.

  • Make story character list.

    Gandalf, Frodo, Sam, Merry, Pippin

  • Split story by events where each event happens in one place, one time and with same characters. Then give event a name.

    Event 1. Starting journey

    Gandalf comes back from a long journey to Frodo and tell him about the One Ring and about black riders. Sam overhear Gandalf's story. Gandalf catches Sam and sends Sam and Frodo to Bree.

    Event 2. Merry and Pippin

    On the way Frodo and Sam meet Merry and Pippin and take them along.

Step 3. Adaptations

On the previous step we make origin events description. At that step we describe event subjective vision for each event participant.

  • Event 1. Starting journey - Sam

    I pollarded bushes and suddenly heard voices of Frodo and Gandalf (…)

  • Event 1. Starting journey - Frodo

    Gandalf came back from a long journey and he was very anxious. (…)

Step 4. Making handouts

On making handouts we gather all adaptations for each character, make final proofreading and make export.

Sam

Story: Ring keeper

  • Event 1. Starting journey

    I pollarded bushes and suddenly heard voices of Frodo and Gandalf (…)

  • Event 2. Merry and Pippin

    We were on acres when suddenly we heard rackets and see two loons (…)

SMTK NIMS workflow

Resume

  • Goal is reached. Handouts are ready.

  • SMTK NIMS is based on described approach. Handouts preparing process is splitted by simple steps and has handy presentation.

  • Moreover there are several secondary LARP tasks and NIMS has tools for them too. Second part of this presentation enumerates these tools.

Advanced handouts tools

Inventory Lists

  • Inventory List - list of items which character should have on game start. First of all these items are prepared by masters.
  • Inventory is related to story. In this case we know where item come from and it is useful in switching character case.
  • Example
    • In story "Starting journey" Frodo gets Sting and mail shirt from Bilbo.
    • In story "Fellowship of the Ring" Frodo gets Galadriel's phial.
  • If some inventory is not related to story you can list it in character profile.

Relations

  • Relation is an information what characters think about each other. It is implemented as text description. Character may have relation both to well known characters and to unknowns. Relation can be unidirectional and bidirectional. We consider characters as familiars if both of them were in one event.

Character profile

  • Character profile - list of important game facts about character.
  • For each character we can fill profile. We use profile to store basic character facts: sex, family, age, abilities, profession, etc. Profile structure is flexible. You can easily add, modify or remove profile fields at any time.
  • Field types:
    • Text - stores text information. Example: character bio.
    • String - stores string information. Example: character motto.
    • Single choice - stores enumeration and one value is selected. Example: sex - male,female,doesn’t matter.
    • Number - number. Example: character age.
    • Checkbox - yes/no value. Example: is character a hacker.
  • Profile is the basis of character filter and groups. Also profile information can be exported in handouts.

Character filter - search by profile

  • If you have many characters you need to search appropriate characters for events by some criteria.
  • For example you need to add young single noble to love story.
  • You can filter characters by profile:
    • Age - greater 18
    • Social category - noble
    • Condition - single
  • And sort character list in ascending order.

Groups

  • Group is a subset of characters which have some common characteristic. Group can have several characters, may be empty, may include all characters. Common characteristic is described by character filter.
  • This approach excludes manual editing of group lists. Updating character profile you automatically exclude character from some groups and add him to other. It means if you check "Templar" checkbox in character profile then character automatically became member of "Templars" group.
  • Group may have text which will be printed in handout for each group member.

Server mode

  • Initially NIMS was made for offline. But for big games we need several story masters coordination.
  • For this case we added server mode. There is a common base in the server and several users can work on one game simultaneously (server provides an Internet site for collaborative LARP game making).
  • There is a strict rights distinction between server users. Each character, story or group can be edited only by one master. Master can easily transfer his character, story or group to other user. This approach excludes accidental corrections.
  • We have more then 10 live servers right now. If you need server for your game tell us and we make one for you.

Information view and planning tools

Statistics and Diagrams

  • NIMS calculates next statistics automatically: quantity of stories/characters/events/symbols and general story completeness.
  • NIMS builds histograms of story event number, story character number and detailed event completeness.
  • NIMS builds automatically histograms for profile number items and donut diagrams for single choice and checkbox items.

Social bonds

  • We use simple logic based approach. If two characters participate in one event we assume these characters have social interaction which formed social bond.
  • It doesn't matter either characters have a date or one character accidently bemired other.
  • Both situations form social bonds.
  • Social bonds can be visualized in several types of social networks:
    • Social bonds - character social bond network.
    • Character-presence-story - network of character-story relations.
    • Character-activity-story - network of character-story relations based on activity type.

Event Timeline

  • You can set event time for each event. This is not necessary but it is recommended. If you set event time then you can print events in chronological order in handouts.
  • If event time is not specified then by default event happens on the game start time.
  • You can see events correlation between different stories in timeline.

Story Map

  • Story map is a schema of conflicts and interests of active groups. You need to add to schema active groups, resources and add relations between them.
  • Resources - any objects or object sets which are points of interest for some characters or character groups. The One Ring, canned goods, magic mana, Moria’s mines - all these things are resources.
  • Relations describe conflicts and interests of character groups. Relation can be started only from group and ended on group or resource. Groups are added from existing groups list. Resource are created directly on story map.
  • Story map is filled manually. Story map building can't be automated.

Character activity workload

  • Each character has different activity type in each story. You can mark your activity expectation for each character to see if there are too much action for player or too low.
  • There are 4 activity types:
    • Active - character is active in story. Examples: Frodo, Don Quixote.
    • Follower - character follows somebody but the main target is not important for him. Examples: Sam, Sancho Panza.
    • Defensive - character is running from chasers. Examples: spy in raid, murderer running from police.
    • Passive - character with some information or artifact but without any goals.
  • Character can have several activities simultaneously in one story.
  • Graphical and statistical activity view will help to analyze character workload for the LARP game.

Game event planning

  • Game event planning - information for master helpers about what and where they need to do something.
  • For example in some story helper must bring letter on second game night. Or it is necessary to be a monster each midnight.
  • You print helpers handouts before game as usual handouts. You can prepare several types of handouts for several helpers.
